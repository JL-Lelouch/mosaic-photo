## 1. 产品概述

**产品名**：Mosaic Moments
**Slogan**：Turn every moment into a masterpiece.
**核心玩法**：用户上传一张主图 + 一批（50–500张）小图素材，系统自动将小图拼成马赛克风格的主图。

### 1.1 核心功能范围
- 主图上传：JPG/JPEG/PNG/WebP，≤10MB，≤8000×8000px，超限自动压缩
- 素材图批量上传：50–500张，单张≤5MB，按需自动裁剪
- 马赛克生成算法（颜色匹配、自动调色、防重复、多轮模拟优化）
- MVP阶段：**完全免费**，预览+下载均免费，下载版降低清晰度并叠加网址水印
- 后续付费点：高清无水印下载、更大图片尺寸、更多素材上限、特殊拼图形状（圆形/三角/六边形）、AI图片识别匹配
- 多语言：中文、英文

### 1.2 页面结构
Logo（居左） + 导航（Home / Create / Examples-Cases / FAQ / Pricing / Support）
→ Hero（标题 + Slogan + 可拖动的前后对比Demo）
→ Create 流程（上传主图 → 上传小图 → 生成）
→ Process 使用说明
→ 使用场景（婚礼 / 礼物 / 旅行 / 宠物 / 毕业）
→ Gallery 多场景展示
→ Blog（马赛克照片相关内容，SEO用）
→ Pricing（免费 / 付费档位）
→ 联系方式 / 用户故事 / 功能建议

### 1.3 开发四阶段（来自 README，本计划严格遵循此顺序）
1. 生成静态页面，确认 UI 设计
2. 实现真实功能（打通闭环）
3. 算法修改优化
4. 实现支付功能

### 1.4 定价方案

| 层级 | 价格 | 权益 |
|---|---|---|
| 免费 | ¥0 | 生成马赛克、网页预览、下载版长边≤3000px+网址水印 |
| 单次权益 | $5 | 解锁**这一次**生成结果：高清无水印下载 |
| 月度权益 | $5 / 30天 | 30天内不限次数生成均可高清无水印下载 + 特殊拼图形状（圆/三角/六边形）+ 更高素材/尺寸上限 |
| 永久权益 | $39 | 不限期不限次数高清无水印下载 + 全部特殊形状 + 未来AI图片识别功能 + 更高素材/尺寸上限 |

（付费权益具体范围以 `prd.md` 为准，此处按讨论内容整理；后续调整直接改这张表）

**几点补充建议**（可采纳可忽略）：
1. **单次与月度同价（都是$5）**：理性用户几乎不会选单次，因为月度同价却能不限次数生成——如果这是有意"引导大家直接买月度"的策略没问题；如果想保留一个更便宜的单次入口，可以把单次降到 $2.99–3.99，和月度拉开梯度。
2. **货币本地化**：$5/$39 面向国际用户；支付宝这边建议单独设人民币价位（参考国内同类工具/贴纸应用的常见定价，如单次¥15-20、月度¥15-20、永久¥199-259），而不是简单按汇率折算，更符合本地付费心理，也省得随汇率波动频繁改价。
3. **月度权益的技术实现**：支付宝当面付不支持自动扣款续订，建议把"月度权益"做成**手动续期的30天时长票**——用户付一次$5拿到30天权益，到期需要自己再付一次，而不是自动订阅扣款。体验上差别不大，但收款实现简单很多；以后接入 Stripe 后如果想做真正的自动订阅，再作为增强功能加上。
4. **付费权益的找回问题**：MVP不强制注册，权益目前设计上绑定匿名 `client_id`（浏览器cookie）。这对"单次"影响不大（付完马上下载），但对"月度"/"永久"这种要跨天/跨设备记住的权益有风险——用户换浏览器、清cookie、换设备就可能"丢了"权益。**建议**：购买月度/永久权益时，让用户留一个邮箱（不强制注册账号，只是留联系方式），生成一个权益凭证/兑换码发到邮箱，之后凭邮箱或兑换码可在任意设备"找回"权益。对愿意付$39买永久权益的用户来说，这是基本的信任保障（对应 `orders` 表需要加一个 `email` 字段，见4.4）。

## 2. 总体架构原则

这是一个**个人独立开发者、轻量化运营**的项目，架构设计始终围绕以下几条硬约束：

| 约束 | 对应决策 |
|---|---|
| 不做云对象存储 | 全部文件落在服务器本地磁盘 |
| 不引入数据库服务器 | 用 SQLite（嵌入式文件数据库） |
| 尽量少额外服务/中间件 | 不引入 Redis，任务队列进程内实现 |
| MVP 不强制注册 | 用匿名 `client_id`（cookie/localStorage）代替账号体系 |
| MVP 完全免费 | 输出分级（预览版 / 降清晰度+水印下载版），支付逻辑推迟到阶段四 |
| 服务器部署在香港/海外 | 免国内 ICP 备案，且不影响支付宝等国内支付渠道的调用 |
| 支付渠道未来会变化 | 做统一的支付适配层，具体渠道可插拔替换 |

**重要澄清**：所谓"本地"，指的是**你租用的服务器（VPS）自带的硬盘**，与用户或你自己的电脑无关。用户浏览器上传的文件通过 HTTPS 写入这台服务器的磁盘，处理后再把结果通过 HTTPS 返回给用户下载，全程不会读写任何人电脑上的其他文件。

## 4. 技术选型明细

### 4.1 前端 + 轻量API
- **Next.js（App Router）+ TypeScript + Tailwind CSS**
- SSR/SSG 用于 Blog / 落地页，利于 SEO
- `next-intl` 做中英文路由级 i18n（**建议阶段一就搭好骨架**，避免后期返工文案）
- Create 页面（上传、拖动前后对比Demo、生成进度轮询）用客户端组件

### 4.2 算法内核（已确定：Python）
- Python（FastAPI 或脚本）+ Pillow/NumPy/SciPy
- Node 通过本地 HTTP 调用或子进程调用 Python 服务，文件路径直接传递（同机磁盘，无需网络传输文件）

### 4.3 存储：本地文件系统
```
/data/uploads/{client_id}/{job_id}/main.jpg
/data/uploads/{client_id}/{job_id}/tiles/*.jpg
/data/cache/{client_id}/tile_features.json
/data/results/{job_id}/preview.png
/data/results/{job_id}/download.png
```
- Nginx（或 Next.js 静态服务）直接对外提供 `results/` 目录访问
- **素材缓存（已确认具体策略）**：
  - 覆盖式保留：每个 `client_id` 只保留"最近一次上传"的素材集合，新的一批上传直接覆盖旧的，不做"多套素材库"管理，简化实现也控制磁盘占用
  - 单 client 容量上限：跟单次生成上限一致（≤500张/约2.5GB），不额外累积
  - 全局 `/data/cache` 目录设总量上限（建议 **15GB**，可按实际VPS盘容量调整），接近上限时按最久未访问（LRU）优先淘汰
  - 过期时间：**7天**未访问自动清理（比生成结果的3天略长，因为素材库对应"用户可能过几天回来微调重新生成"的场景）
  - `client_id` cookie 有效期建议设 **30天**，比服务器端数据保留时间长一些无妨，只需在前端做好兜底：素材库已被清理时提示用户重新上传，而不是报错
  - 隐私提示：页面/隐私政策上注明"素材会缓存N天以便您再次编辑，之后自动清理"，让用户知情
- **特征缓存**：素材图的 LAB 平均色/特征值算过一次后缓存（key为文件内容hash），避免重复计算
- **清理策略**：定时任务（`node-cron` / Python `APScheduler`），**3天**清理未下载/未支付的临时文件（已确认，后续可视磁盘占用情况调整），也是对用户隐私（婚礼/宠物/毕业等私人照片）负责

### 4.4 "数据库"：SQLite
无需单独部署维护的数据库服务进程，用一个文件即可：

| 表 | 用途 |
|---|---|
| `jobs` | 任务状态/进度/参数/结果路径/过期时间 |
| `tile_features` | 素材特征缓存（file_hash → LAB均值等） |
| `orders`（阶段四） | 订单记录、支付渠道、支付状态、购买邮箱（用于权益找回，见1.4） |

Blog 内容直接用 MDX 文件放仓库里，走 Next.js 静态生成，不进数据库。

### 4.5 任务队列：进程内实现
不引入 Redis。用有限并发的 worker（线程池/进程池）+ SQLite 记录状态，前端轮询查询进度。MVP阶段并发量可控，足够用；未来流量上来了再引入 Redis/Celery。

### 4.6 部署
- **已确认**：优先复用现有的新加坡 VPS（未到期），跳过国内 ICP 备案，出海访问速度更好，且不影响支付宝等接口调用；如后续这台机器不够用（并发/磁盘不足），再考虑购买阿里云国际版海外节点补充或替换
- 起步阶段先按现有配置跑，观察实际负载后再决定是否需要升配/加盘
- Docker Compose 把 Next.js + Python 算法服务两个容器一起管理，或用 pm2 + systemd 管两个进程

**⚠️ 域名与部署方式的重要说明**：讨论中提到"可能用 Vercel 子域名直接托管"，这里需要提醒一个架构冲突——

Vercel 跑的是 **Serverless/Edge 函数**，没有持久化本地磁盘，也不能常驻运行 Python 算法服务，这跟我们"本地文件系统 + SQLite + 常驻 Python worker"的整套设计是不兼容的（部署到 Vercel 上，每次请求可能落在不同的临时实例上，写入的文件、SQLite 数据下一次请求就可能读不到了）。

所以结论是：**Next.js 前端也必须和 Python 算法服务部署在同一台常驻服务器上**（也就是这台新加坡 VPS），而不能用 Vercel 托管前端。域名方面：
- 短期最简单：先不买域名，直接用 VPS 公网 IP 访问（缺点：没法用 Let's Encrypt 免费 HTTPS 证书，体验和SEO都打折扣）
- **建议**：花几十元买一个域名，DNS 解析指向这台新加坡 VPS 的 IP，配合 Let's Encrypt 上 HTTPS，成本很低但换来一个统一、可控、符合整体自建架构的方案；具体候选域名对比与推荐见第9节「决策记录」

### 4.7 匿名身份识别
用 cookie/localStorage 存一个 `client_id`（UUID），服务端以此归类该访客的上传素材、历史生成记录和临时文件目录，不需要注册登录即可支持"素材复用"体验。

## 5. SEO 实现方案（分板块，阶段一即落地）

**核心原则**：SEO 基础设施要跟着阶段一的静态页面一起搭骨架，不要"先做页面、以后再补"——和 i18n 一样，页面结构/路由方式定下来之后再改，返工成本很高。自托管 Next.js（而不是 Vercel）依然完整支持 SSR/SSG/ISR，不影响 SEO 能力。

### 5.1 技术层面（贯穿所有页面）
- SSR/SSG 渲染所有营销页（Home/Blog/Gallery/FAQ/Pricing/使用场景），博客和Gallery可用 ISR 定期静态再生成
- 每个页面独立的 `title`/`meta description`/`canonical`/Open Graph 标签（`og:title`/`og:description`/`og:image`），马赛克照片是强分享型内容，OG标签直接影响社交分享时的展示效果
- 结构化数据（JSON-LD）：FAQ模块用 `FAQPage`、Process模块用 `HowTo`、每篇博客用 `BlogPosting`、Gallery图片用 `ImageObject`、Pricing用 `Product`/`Offer`
- `next-sitemap` 自动生成 sitemap.xml + robots.txt；具体某个用户生成结果页（带 `job_id` 的私人马赛克图）设置 `noindex`，避免大量临时/低质页面被收录
- 中英文 i18n 的 `hreflang` 互相声明，URL按语言区分（如 `/en/...` `/zh/...`），避免多语言重复内容问题
- 图片优化：Next.js `Image` 组件、WebP/AVIF、响应式尺寸、懒加载
- Core Web Vitals：营销页尽量减少非必要客户端JS，保证首屏渲染速度

### 5.2 按板块的具体策略
- **Home/Hero**：核心关键词布局（如 "photo mosaic maker"／"马赛克照片生成器"），H1 包含关键词
- **Create 工具页**：交互控件本身不能被抓取，页面上要有清楚的说明性文字（这是什么、怎么用），否则Google拿不到任何可理解的内容
- **Process 使用说明**：适合做 `HowTo` 结构化数据，有机会拿到步骤富摘要
- **使用场景（婚礼/礼物/旅行/宠物/毕业）**：**建议每个场景单独做成一个落地页**（而不只是首页里的一个区块），针对"婚礼马赛克照片礼物""宠物照片马赛克拼图"这类长尾词，比硬抢"photo mosaic"这种大词容易见效得多，是这类产品投入产出比最高的SEO点之一
- **Gallery**：每张作品配详细的 `alt` 文本和 `ImageObject` 结构化数据，争取 Google 图片搜索的流量（视觉型产品，图片搜索本身就是不小的入口）
- **Blog**：长期获客主力，围绕"如何制作马赛克照片""XX节日马赛克礼物创意"等信息型长尾词做内容库，每篇内部链接回 Create/Pricing/对应使用场景页
- **FAQ**：`FAQPage` 结构化数据，容易拿到手风琴富摘要，几乎零成本增加曝光

## 6. 马赛克算法设计

1. **主图网格切分**：根据输出分辨率和素材数量，自动计算合适的格子大小
2. **颜色特征提取**：主图每格、每张素材图都转换到 **LAB 颜色空间**计算平均色（比RGB更符合人眼感知）
3. **最近邻匹配**：用 KD-tree（素材≤500张，暴力比较也很快）为每个格子找若干颜色最接近的候选素材
4. **防止连续重复**：从候选中按"最近若干格子是否已用过"做筛选/加权随机
5. **自动调色**：颜色误差较大时，对选中素材做亮度/对比度/色温微调
6. **多轮模拟优化**：初始分配后计算整体误差，做几轮局部调整（不同网格密度/交换相邻格子素材），选效果最好的一版
7. **输出**：高分辨率 PNG，可选叠加一层半透明主图
8. **付费能力预留位**：
   - 特殊形状拼图（圆形/三角/六边形）：切分方式从矩形网格换成对应镶嵌方式，输出时按形状做 mask 裁剪
   - AI图片识别：可插拔的匹配策略，未来接入类似 CLIP 的语义级匹配

**输出分级**（对应 MVP 免费策略）：
- 预览版：正常清晰度网页展示
- 下载版（免费）：降采样至**长边≤3000px**（已确认，具体效果待实际生成验证，后续可调整）+ 右下角叠加半透明网址水印
- 下载版（付费，阶段四）：原始分辨率 + 无水印

## 7. 支付适配层设计（阶段四）

统一抽象接口，业务代码只依赖接口，具体渠道可插拔：

```
PaymentProvider（抽象接口）
 ├─ create_order(order_id, amount, currency)  → 返回该渠道的支付方式（二维码/跳转链接/收款地址）
 ├─ check_status(order_id)                    → 查询订单是否已支付
 └─ handle_webhook(request)                    → 统一转换各渠道异步通知为"订单已支付"事件
```

| 渠道 | 状态 | 说明 |
|---|---|---|
| 支付宝 | **先实现** | 官方接口，`create_order`生成二维码/链接，`handle_webhook`接支付宝异步通知验签 |
| Stripe | 接口占位 | 方法签名先写好，内部先返回"暂未开放"；账号批下来后只需填充实现，业务层不用改 |
| TRC20 (USDT) 等链上收款 | 待实现 | 方案A（推荐）：金额尾数唯一化绑定订单，定期调公开API（TronGrid/Tronscan）核对到账；方案B：用户提交TXID，后台验证 |

收款地址等敏感信息放 `.env` 配置文件，不进代码库，后续替换为你的个人地址只是改配置、不涉及代码改动。

> 提醒：加密货币收款如涉及后续换汇入境内银行账户，建议自行了解清楚相关合规要求；本文档不构成法律/财税建议。

**国内外收款渠道现实情况备忘**（供后续参考，政策变化快，以官方最新信息为准）：
- Stripe 目前对中国大陆个人身份基本无法直接申请，需海外公司主体或美国个人税号
- 长期更正规路径：注册个体工商户 → 申请官方支付宝/微信商户号（覆盖国内）+ 跨境聚合支付如PingPong/连连/Airwallex（覆盖国际卡）
- 零资质过渡方案：支付宝当面付（限额：单笔≤1000元/日≤5万元）覆盖国内 + Gumroad（Merchant of Record）覆盖海外卡/PayPal
- 不建议：所谓"个人免签支付"灰色工具，存在账户冻结/资金安全风险

**已确认的实施节奏**：
- 前期直接用**支付宝个人（当面付）**收款，预期订单量不大，在限额内可以接受，暂不急着办个体工商户
- 待后续订单量明显增长后，再评估是否办理个体工商户以升级到正规商户号、扩展更多渠道
- Stripe 暂不申请，等海外订单量起来后再申请（当前 `StripeProvider` 只做接口占位）

| 事项 | 结论 |
|---|---|
| 算法内核语言 | **Python**（FastAPI/脚本 + Pillow/NumPy/SciPy） |
| 个体工商户 | 前期用**支付宝个人当面付**（预期订单量不大，限额内可接受），订单量增长后再评估升级 |
| VPS | 复用**现有新加坡服务器**（未到期），不够用再买阿里云海外节点补充/替换 |
| 域名候选对比 | 见下方「域名选择」，推荐 **mosaicmoment.net** |
| 免费下载版分辨率上限 | 暂定 **长边≤3000px**，效果待验证，可调整 |
| 临时文件保留天数 | **3天** |
| 素材库缓存策略 | 覆盖式保留，单client≤500张/2.5GB，全局配额15GB(LRU淘汰)，7天未访问清理，cookie有效期30天（详见4.3） |
| 定价方案 | 免费 / $5单次 / $5-30天 / $39永久，详见1.4，含人民币本地化与权益找回建议 |
| Stripe 申请时机 | 暂不申请，等海外订单量起来后再申请 |

### 域名选择：mosaicmoment.vip / .net / .xyz / .top

四个候选里，**推荐 `mosaicmoment.net`**，理由：

- 这几个 TLD 对 SEO 排名本身没有直接影响（Google官方说法是不把TLD作为排名因子），差异主要体现在**用户信任度**和**邮件送达率**两方面——而这个项目后续要收款、要给用户发权益找回邮件，这两点都很重要。
- `.xyz` 和 `.top` 属于邮件发送场景里被明确标记为"高滥用/需谨慎"的后缀——行业数据显示 .xyz 域名的滥用率大约在10-15%，导致邮件服务商对这类发件域名普遍采取更严格的过滤，而 .com、.net、.org 属于邮件送达率最高的第一梯队；也有资料把 .xyz 和 .top 一起列入"不建议用于正式邮件发送"的后缀名单。对一个要发支付收据、权益找回邮件的产品来说，这个风险不太划算。
- `.vip` 没有被上述资料列为问题后缀，作为备选可以接受，且"VIP"的含义还算贴合付费档位的定位，但整体知名度和信任积累不如 `.net`。
- 小提醒：品牌名是复数的 "Mosaic **Moments**"，候选域名是单数 "mosaicmoment"，属于小的不一致，不影响使用，但注册时可以顺手看看 `mosaicmoments.net`（带s）是否也能拿到，两个一起指向同一站点，防止别人抢注造成混淆。
